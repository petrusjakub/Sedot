# Multi-format social media downloader with Cobalt proxy worker

The single-file downloader that previously grabbed only the first video from a link was rewritten to support three download modes (video, audio, muted video), five quality presets, and a no-watermark toggle across seven platforms (TikTok, Instagram, Facebook, YouTube, Pinterest, Threads, Shopee). A new Cloudflare Worker (`worker.js`) acts as a CORS proxy to the Cobalt API, forwarding format and quality parameters. The frontend now properly handles Cobalt's `picker` response type, displaying all carousel/slideshow items instead of silently taking only the first.

**Watch for:** The no-watermark toggle is wired in the UI but never sent to the worker or Cobalt API (confirmed). The CORS policy is `Access-Control-Allow-Origin: *` with no origin restriction or rate limiting, making the worker an open relay for any origin (confirmed). XSS via thumbnail URLs injected into innerHTML is possible when Cobalt returns attacker-controlled `thumb` values (likely).

## High-level view

The frontend went from a simple URL-in/link-out flow to a format-aware multi-result UI. Format selection maps to Cobalt's `downloadMode` parameter (`auto`/`audio`/`mute`), quality maps to `videoQuality`, and response handling branches on `status` (`redirect`/`tunnel`/`picker`). The picker branch renders each item with its thumbnail and a per-item download button, which is the major UX upgrade over the old "grab first item" approach.

The worker is a thin proxy: validate that `url` exists, assemble the Cobalt request body with sensible defaults, forward to `https://api.cobalt.tools/`, and pipe the JSON response back with CORS headers. It has no authentication, no origin allowlist, and no rate limiting — acceptable for a personal tool but would need hardening before any public exposure.

The no-watermark toggle (`#noWatermark`) exists purely as decoration. The checkbox state is never read by `buildRequestBody()` and no corresponding parameter is sent to Cobalt. Cobalt's default behavior already strips watermarks where possible, so the feature "works" by coincidence for TikTok, but the UI is misleading — toggling it off does nothing.

Shopee is listed as a supported platform, but Cobalt's public documentation does not list Shopee as a supported service. Links from `shopee.co.id` or `shopee.com` will pass the frontend's platform validation but will likely return `error.api.link.invalid` from Cobalt.

## Disconnected no-watermark toggle

`buildRequestBody()` constructs the request payload at line ~285 of index.html:

```javascript
function buildRequestBody(url) {
    const downloadMode = document.getElementById('downloadFormat').value;
    const videoQuality = document.getElementById('videoQuality').value;

    return {
        url: url,
        downloadMode: downloadMode,
        videoQuality: videoQuality,
        filenameStyle: 'basic',
        tiktokFullAudio: true,
        tiktokH265: false,
        disableMetadata: false
    };
}
```

The `noWatermark` checkbox is never queried. Cobalt doesn't have a single "remove watermark" boolean — for TikTok, watermark removal is the default behavior. But if the toggle is presented to users, its off-state should do something (e.g., not pass `tiktokFullAudio`, or add a watermark-preserving parameter if one exists). As-is, the toggle is pure theater.

## XSS surface in picker rendering

`showPickerResults` builds HTML via string concatenation and assigns it to `innerHTML`:

```javascript
const thumbHtml = item.thumb
    ? '<img src="' + item.thumb + '" alt="Thumbnail" class="result-thumb">'
    : ...;
```

If Cobalt ever returns a `thumb` value like `" onerror="alert(1)"`, or if the worker is pointed at a different backend, this becomes a stored-XSS vector. The same applies to `item.url` being placed in an `href`. Using `document.createElement` and setting `.src`/`.href` as properties (which don't interpret HTML) would close this.

## Worker as open relay

The worker validates only that the request is POST and that `body.url` is truthy. Any website on the internet can call this worker and use it to proxy requests to Cobalt. This doesn't expose user data, but it does expose the worker to abuse: an attacker could use it as a free Cobalt proxy at the owner's Cloudflare billing cost.

The minimal fix is checking `request.headers.get('Origin')` or `request.headers.get('Referer')` against an allowlist and returning 403 for unrecognized origins. A more robust approach would be a pre-shared token in a custom header.

## Cobalt API compatibility

The worker sends requests to `https://api.cobalt.tools/` with `Accept: application/json` but no `Authorization` header. Cobalt's API v7+ requires authentication for most instances. If the public instance at `api.cobalt.tools` is still accepting unauthenticated requests, this works — but it may break without warning when the instance operators enforce auth. The worker's `env` parameter is available and could read a `COBALT_API_KEY` secret from Cloudflare environment variables.

## Platform detection vs. actual support

The frontend validates URLs against seven platform patterns. Six of these (TikTok, Instagram, Facebook, YouTube, Pinterest, Threads) are supported by Cobalt. Shopee is not. A user pasting a Shopee product video URL will see "Mengekstrak konten dari Shopee..." followed by an error message like "Link tidak valid" — a confusing experience that implies the link is wrong rather than the platform being unsupported.
